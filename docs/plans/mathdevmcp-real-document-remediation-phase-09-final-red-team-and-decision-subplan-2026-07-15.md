mathematical backend, changes no target or assumption, reruns no CAS/proof
search, and makes no source edit. A discovered defect is classified before any
repair is considered.

### Candidate, Review, And Finalization Sequence

1. `create-candidate` allocates its immutable literal run root before
   reconstruction and writes the reconstruction, adversarial, reconciliation,
   run-manifest, and immutable `candidate-decision.json` for every first-run
   outcome. A readable evidence/claim-boundary mismatch is durably `UNSAFE`; a
   genuine environment, authority, execution-guard, or unresolved runner
   failure is durably `BLOCKED`; neither may disappear into stderr-only state.
   It does not write `decision.json` or claim mission completion.
2. `verify-candidate` requires the literal candidate digest returned by create
   and independently checks the candidate artifact inventory,
   candidate decision digest, predecessor before/after inventories, zero live
   mathematical-backend/document-audit/network execution count, exact current
   reviewed code bindings, and fixed authority fields.
3. A fresh local Codex agent performs one substantive read-only result review
   over the literal candidate root, runner, tests, and draft result. The review
   cannot edit artifacts or authorize publication, defaults, release, or source
   changes. Its durable record contains exactly one literal binding line for
   each of the run root, candidate semantic digest, candidate file SHA-256, and
   candidate artifact-inventory digest, followed by an exact terminal verdict.
4. Codex writes a review adjudication and invokes `finalize` exactly once. The
   finalizer revalidates candidate, predecessor, and reviewed code state before
   and after staging adjudication. Each artifact becomes visible only after a
   complete fsynced no-overwrite write. If interrupted after adjudication but
   before `decision.json`, retry is allowed only when the staged adjudication is
   byte-identical to the same candidate, review, and outcome; changed state is
   rejected. `decision.json` remains no-overwrite and single-use. Final
   verification requires the literal digest returned by finalization.

Review outcomes are fixed:

| Review outcome | Final treatment |
| --- | --- |
| `AGREE` with no material finding | Retain the evidence-derived candidate status. |
| Accepted claim-boundary, evidence-integrity, privacy, or publication finding | Final status is `UNSAFE`. |
| Review unavailable, review result malformed, or an unresolved review/runner engineering issue prevents trustworthy classification | Final status is `BLOCKED`. |

A disputed material finding cannot be silently rejected. Retaining a safe
candidate after such a finding requires a focused read-only rereview that
returns `AGREE` without changing candidate bytes. Code or evidence repair after
`create-candidate` requires a new amended run from the earliest invalidated
phase.
