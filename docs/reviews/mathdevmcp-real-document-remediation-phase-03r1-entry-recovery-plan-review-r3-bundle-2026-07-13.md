## Recovery Mechanism

The recovery bootstrap accepts only exact clean-environment `audit`,
`preflight`, or readiness-bound `create` invocations under CPython 3.11.15
with `-B -S`. It preserves the base runtime measurement, 17 ordered P02
obligation bindings with state counts 14/2/1 and 14 eligible, exact P03 action
registry, implementation allowlist, and publication-disabled state.

Only these roots are excludable:

```text
.local/mathdevmcp/evidence/p02r3-20260712/result-rounds/rr01/governance/tmp
.local/mathdevmcp/evidence/p02r3-20260712/result-rounds/rr02/governance/tmp
.local/mathdevmcp/evidence/p02r3-20260712/result-rounds/rr03/governance/tmp
```

For each root, descriptor-relative `O_NOFOLLOW` traversal freezes 151 regular
files, 69 directories including root, 12 symlinks, 74,331 regular-file bytes,
and an ordered full inventory digest. Every symlink target must be absolute,
inside its own root, and an independently inventoried regular file/directory.
Relative, dangling, escaping, special, count-drift, byte-drift, or inventory-
drift cases fail.

Before inventory, `_verify_scratch_registry()` enumerates direct
`result-rounds/<round>/governance/tmp` nodes with descriptor-relative no-follow
opens and requires the discovered set to equal the three literals. During the
later formal-tree walk, `_is_p02r3_scratch_namespace()` independently rejects
any matching unregistered root, including a fourth tree appearing after the
registry scan. The walker skips only an exact registered directory node and
rejects every other predecessor symlink or special path.

All formal P00-P02R3 regular files outside the three roots enter the protected
manifest and are scanned for logical or absolute references to excluded roots;
any reference fails. Recovery history files that necessarily describe those
roots are protected by exact digests instead of this text scan. Scratch content
is never formal evidence, and its exclusion does not validate its test output.

`_prepare()` performs root/history/runtime/P02/review/budget checks, all input
tree discovery, all manifests, and all five payload constructions. Audit omits
review/budget authority and writes nothing. Preflight requires the exact R3
`AGREE` and canonical carry-forward authority and writes nothing. Both expose a
canonical readiness projection; create reruns authority-bound `_prepare()` and
requires its exact digest in argv before `_allocate()`.

Readiness is deliberately two-level to avoid a self-reference. A canonical
preparation projection binds authority, predecessor/runtime state, reference
counts, and the four non-record payload digests; its digest and the final
readiness schema identifier enter `entry-record.json`. The final readiness
projection then adds that preparation digest and the serialized entry-record
digest, thereby binding all five exact payloads before allocation. The final
digest is emitted by preflight/create and preserved in the execution ledger,
not embedded recursively in the record it hashes.

Before `_mkdir_entry`, `_allocate()` independently requires the exact five
payload keys, rehashes every prepared payload against the final-readiness
fields, and reconstructs the canonical final projection. Changed, missing, or
extra prepared bytes fail while the P03 root is absent.

After `_prepare()`, `_allocate()` only creates fixed output directories, writes
the five prepared bytes with no-replace operations, reopens those fixed paths,
and enumerates the new output directories for exact shape. A focused test
poisoned every prepare-time discovery helper and allowed `_read` only for the
five outputs; allocation passed. The plan explicitly assumes a supervisor-
controlled single-writer create window and stops if that cannot be maintained;
it does not claim an adversarial atomic filesystem snapshot.

The five immutable entry files are the implementation, protected, and
immutable manifests, `scratch-exclusion-ledger.json`, and `entry-record.json`.
The entry binds base/recovery/failure/review/budget history, P02 state, runtime,
allowlist, actions, non-record payload digests, preparation digest, readiness
schema, CPU-only mode, and publication disabled.
