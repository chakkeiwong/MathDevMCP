## Phase 09 Baseline Amendment

The first real calibration rung found a material baseline error not visible in
the pre-execution records. `566 extractable labels` meant indexed equation rows
after excluding seven nested-ownership labels, not 566 adapter-eligible
canonical obligations. Exact full-session resolution on the frozen source found
434 canonical adapter targets and 132 exact source-bound relation-shape
abstentions.

The repaired execution contract is:

- audit/fix and rigor account for all 566 labels, preserving the 132 typed
  abstentions as first-class records;
- derivation-tree execution covers 434 canonical targets and carries an exact
  132-label exclusion/abstention ledger;
- no quarantined relation is silently dropped or promoted to a tree target.

This amendment strengthens rather than relaxes the original evidence contract
that every requested label receive either one validated record or one explicit
typed status.

## Material Findings

| Finding | Why material | Plan repair |
| --- | --- | --- |
| A shell-level label chunker would generate independent proposal reports | Proposal synthesis has cross-label and document-level ledgers; concatenating reports is not equivalent | Persist proof-audit records, then invoke the existing document synthesis once over the complete ordered record set |
| Passing only an index-cache path does not remove repeated parser work | `audit_derivation_v2_for_label` calls parser-policy selection per label and the current parser rebuilds the index | Require session-bound validated index/parser context or equivalent injection before claiming the scale gap closed |
| Increasing workers was initially tempting | Process/thread parallelism can multiply index parsing, backend initialization, memory, and nondeterminism | Use serial execution as the correctness baseline; calibrate concurrency only after resumability/equivalence passes |
| One generic checkpoint schema would conflate proof audits and tree targets | Tree records contain semantic packets, branches, blockers, rankings, compiler output, and promotion vetoes absent from proof audits | Define separate per-label proof-audit and per-target tree schemas under one source/session identity |
| Batch completion alone could become the pass metric | Successful process exit does not establish exact coverage or semantic equivalence | Make identity, coverage, mutation rejection, and uninterrupted/resumed semantic equivalence the primary gates |
| Runtime thresholds lacked provenance | Arbitrary thresholds can reject correct work or excuse unbounded work | Treat 8/10-minute bounds as engineering hypotheses calibrated on 1/4/8/16-label rungs, never scientific criteria |
| Packet reuse could bypass source validation | Reusing a convenient audit result could cross versions or configurations | Require exact session, label, source, obligation, options, and contract validation before packet construction |
| Nested-label repair could overclaim parser authority | LaTeXML agreement is not mathematical ownership proof | Use external parsers diagnostically and promote only one unambiguous byte/source-span ownership rule; otherwise preserve abstention |
| Scope repair could turn syntax into semantic assurance | Static function signatures cannot prove implementation correctness | Add explicit classification states and retain a non-proof boundary |
| D447 might again be described as generalization evidence | D447 was produced after MathDevMCP-assisted D446 repair | Keep D446/D447 contaminated regression/capstone sources and require externally clean provenance for a future holdout |

## Feasibility Review

- Existing atomic no-replace artifact utilities can support immutable records.
- Existing proof-audit results are the correct reusable unit for audit/fix and
  packet consumers.
- Existing audit/fix synthesis already accepts evidence but needs an exact
  precomputed-label path with complete coverage semantics.
- Existing rigor reuse validates file, digest, labels, and completeness and can
  be extended to session/aggregate identity.
- Existing tree execution already isolates per-target work internally, so it
  can add target checkpoints while retaining the current document assembler.
- Existing document-response artifacts and cursor resolution remain downstream
  presentation machinery; they do not by themselves provide computation
  resumability.

## Boundary Review

- DynareMCP remains read-only.
- Publication remains disabled.
- No default/release change is authorized.
- Backend absence is diagnostic, not a mathematical failure.
- The plan does not authorize source-document repair.
- Independent review is required before any future mathematical/product claim
  promotion, but ordinary implementation phases may proceed under the active
  academic governance reset.

## Review Decision

The repaired plan has correct baselines, explicit evidence and nonclaim
contracts, feasible artifact boundaries, mutation tests, bounded scale rungs,
and valid stop conditions. Execution may proceed.
